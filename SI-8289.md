In certain cases, the html files generated by Scaladoc may contain sections that vary non-deterministically. When compiling the compiler with itself, the generated documentation can be therefore different each time; an extended "stability" check that also verifies the produced documentation will therefore fail. An example is enclosed. The diff is:
{code:scala}
61c61
<             <div class="subClasses hiddenContent"><a href="JavaUniverse.html" class="extype" name="scala.reflect.api.JavaUniverse">JavaUniverse</a>, <a href="../macros/Universe.html" class="extype" name="scala.reflect.macros.Universe">Universe</a>, <a href="Universe.html" class="extype" name="scala.reflect.api.Universe">Universe</a></div>
---
>             <div class="subClasses hiddenContent"><a href="JavaUniverse.html" class="extype" name="scala.reflect.api.JavaUniverse">JavaUniverse</a>, <a href="Universe.html" class="extype" name="scala.reflect.api.Universe">Universe</a>, <a href="../macros/Universe.html" class="extype" name="scala.reflect.macros.Universe">Universe</a></div>
{code}
Here there are two "Universe" classes; probably during sorting one of the two is picked at random (the qualified name could be used for sorting instead).
The non determinism in the example above can be fixed by using the following patch:
{code}
--- a/src/scaladoc/scala/tools/nsc/doc/html/page/Template.scala
+++ b/src/scaladoc/scala/tools/nsc/doc/html/page/Template.scala
@@ -652,7 +652,7 @@ class Template(universe: doc.Universe, generator: DiagramGenerator, tpl: DocTemp
           <div class="toggleContainer block">
             <span class="toggle">Known Subclasses</span>
             <div class="subClasses hiddenContent">{
-              templatesToHtml(subs.toList.sortBy(_.name), scala.xml.Text(", "))
+              templatesToHtml(subs.toList.sortBy(_.qualifiedName.split("\\.").reverse.mkString(".")), scala.xml.Text(", "))
             }</div>
           </div>
         else NodeSeq.Empty
{code}

Not terribly important, but it would be nice to have determinism in Scaladoc as well.
https://github.com/scala/scala/pull/4889 will fix the order of subclasses.

In addition to that and val_N, GraphViz's SVG generation is non-deterministic. A SVG output is different sometimes even the input is the same.
